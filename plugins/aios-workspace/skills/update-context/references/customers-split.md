# customers 分割戦略

## いつ分割するか
以下のいずれかに当てはまる場合に分割する：
- ファイル全体が **300行を超える**
- 把握している顧客（個人名・クライアント名）が **5件以上** ある
- 各顧客について **10行以上** の記述がある（取引履歴、関係性、特性など）
- 「顧客から聞こえること」が **15項目以上**、または日付付きの履歴を残したい

## ターゲット構造
```
context/customers.md           ← インデックス（要約集）
context/customers/
├── <client-name>.md           ← 個人名・クライアント名ごとの詳細
├── icp.md                     ← ICPの長文記述
└── feedback.md                ← フィードバックアーカイブ
```

クライアント名で分けるのが基本。セグメント別ではない。同じセグメントに属する複数クライアントがあっても、それぞれ独立したファイルにする。

## インデックスファイルに残すもの

インデックスは「目次」ではなく「**要約集**」として機能させる。各サブセクションは **2〜3文の短いサマリー** を持ち、その末尾にサブファイルへのリンクを添える。読者がリンクを開かなくても全体像が掴めるレベルに保つ。

- **各クライアント** — 2〜3文のサマリー（誰か、いつから取引、規模・取引額、関係の特徴やステータス）+ サブファイルへのリンク
- **ICP** — 2〜3文のサマリー（理想顧客の核心的特徴）+ リンク
- **フィードバックの主要テーマ** — 2〜3文で要約（最頻出のフィードバック傾向）+ アーカイブへのリンク
- **主要な獲得チャネル**（短いので本文のまま）
- **セグメント分類**（任意） — 顧客をグルーピングするカテゴリがあれば1〜2文で言及（例:「SMB中心、エンタープライズ案件が増加傾向」）

## サブファイルに移すもの
- 各クライアントの詳細プロフィール（業界、規模、担当者、契約内容、取引履歴、特記事項、関係性のメモ）
- ICPの長文記述、判定基準
- フィードバックの整理されたアーカイブ（日付付き、クライアント名付き）

## 命名規則
- クライアント: `<kebab-case-client-name>.md`
  - 例: `acme-corp.md`、`tanaka-shoten.md`、`yamada-taro.md`
  - 個人名の場合は氏名のローマ字または日本語をそのままkebab化（プライバシーに注意）
  - 同名衝突回避のため、必要なら接尾辞や業界を付ける（例: `acme-corp-jp.md`、`tanaka-construction.md`）
- ICP: `icp.md`
- フィードバック: `feedback.md`

## 分割実行時の注意
- インデックスからサブファイルに内容を「移す」のであり、両方に書かない（重複禁止）
- インデックスのリンクは相対パス（例: `[詳細](customers/acme-corp.md)`）
- クライアントファイルには取引開始日、現在のステータス（アクティブ/休眠/解約）、最終接触日を必ず含める
- インデックスの先頭に `Last updated:` 行を残す
- 個人情報を含むため、リポジトリを公開する場合は `.gitignore` で `context/customers/` を除外することを検討
