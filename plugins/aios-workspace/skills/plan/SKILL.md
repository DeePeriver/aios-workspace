# Plan

行動の前に書面の計画を作成する。

## 手順

### 1. リクエストを理解する
目標が不明確な場合は最大3個まで明確化質問をする。すでに明確な場合は質問しない。よくある明確化質問：
- 締め切りはあるか（あればいつか）？
- あなたにとっての成功とは何か？
- 予算 / リソース制約は？

### 2. 関連するコンテキストを引き出す
当てはまるコンテキストファイルを読む：
- マーケティング / セールス / ブランド → `customers.md`、`competitors.md`、`strategy.md`
- 採用 / オペレーション / ベンダー → `personal-profile.md`、`financials.md`、`current-priorities.md`
- 価格設定 → `customers.md`、`competitors.md`、`financials.md`
- プロダクト方針 → `customers.md`、`strategy.md`
- 戦略的決定 → `strategy.md`、`financials.md`、`current-priorities.md`
- 財務関連 → 必ず `financials.md` を含める

すべてを読まない。当てはまるものだけ読む。

### 3. 計画をドラフト
この形式を使い、`plans/YYYY-MM-DD-<short-slug>.md` に保存する。slugは2〜4語のkebab-case記述。

```markdown
# <計画タイトル>

**作成日:** <今日の日付>
**ステータス:** Draft

## コンテキスト
<2〜4文：状況、なぜ今、どのコンテキストファイルが情報源か>

## ゴール
<1〜2文：成功とは何か、可能なら測定可能な成果を含める>

## アプローチ
<2〜4文：高レベルの戦略 — 代替案ではなくこのアプローチを選ぶ理由>

## ステップ
1. <アクション> — <期待される成果物 / アウトプット>
2. <アクション> — <期待される成果物 / アウトプット>
3. ...

## 成功基準
- <測定可能な成果>
- <測定可能な成果>

## リスク・未解決の質問
- <リスク> → <緩和策>
- <解決すべき未解決の質問>
```

### 4. 保存前に見せる
計画をユーザーに見せる。これで合っているか尋ねる。フィードバックに基づいて、保存前に反復する。

### 5. 保存して報告
承認されたらファイルを保存する。パスをユーザーに伝え、次のステップとして `/aios-workspace:execute <path>` を提案する。

## ガードレール

- 計画は決断して行動する経営者向けであり、プロジェクトマネージャー向けではない。ステップは具体的かつ少なく保つ。
- 計画のステップが7個を超えると、それはおそらく2つの計画 — 分割する。
- 決定や成果物のないリサーチのみのステップは含めない。リサーチが必要な場合、成果物を明示する（例：「X、Y、Zの比較によりベンダーを決定」）。

## やってはいけないこと

- ユーザーに見せる前に計画を保存しない。
- ふわっとしたマネジメント用語を書かない。平易な言葉が勝つ。
