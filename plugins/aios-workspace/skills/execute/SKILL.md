# Execute

計画を進めて成果物を生み出す。

## 手順

### 1. 計画の場所を特定
- ユーザーがパスや名前を指定した場合は、それを使う。
- 指定がない場合は `plans/` の最近のファイルを一覧表示してどれを実行するか尋ねる。
- 計画のステータスがすでに `Complete` の場合、再実行するかを確認する。

### 2. 計画を最後まで読む
始める前に計画全体を読む。ゴール、アプローチ、成功基準を内面化する — 実行中の判断はそれらに従う。

### 3. 各ステップを実行
すべてのステップを順番に：

1. **宣言する** どのステップに取り組むかを1行で。
2. **作業する** — リサーチ、ドラフト、計算、比較、アウトリーチ準備など、ステップが要求するもの。
3. **成果物を保存する** `outputs/<plan-slug>/<step-name>.<ext>` に。
   - 書面の成果物には `.md` を使う（ほとんどの場合）。
   - `outputs/<plan-slug>/` ディレクトリがなければ作成する。
4. **ステップを完了マーク** 計画ファイルで：行頭の `1.` を `1. ✓` に変更する。
5. **次のステップ前に確認の一時停止** ステップにユーザーから見えるインパクトがあった場合（レビューすべきドラフト、決定すべき事項、検証すべき数字）。

### 4. 計画ステータスを更新
すべてのステップが完了したら：
- `ステータス: Draft` → `ステータス: Complete` に変更
- `作成日:` の下に `完了日:` 行を追加して今日の日付を記入

### 5. 報告
サマリー：
- 何が生み出されたか
- すべての出力ファイルへのパス
- 成功基準が満たされたか
- 残った未解決事項

## 一時停止してユーザーに確認すべき時

- ステップが手元にない情報を要求する場合。推測しない。
- ステップの成功基準が手元のリソースで満たせない場合。フラグして計画の修正を提案する。
- 出力がビジネスをセンシティブな何か（金銭的コミットメント、対外コミュニケーション、採用決定、顧客向けメッセージ）にコミットさせる場合。ドラフトを見せ、最終保存の前に明示的承認を要求する。

## やってはいけないこと

- ステップを黙ってスキップしない。当てはまらなくなった場合は `~~取り消し線~~` を入れて1行の理由を添える。
- 1つも見せずにすべての成果物を書き上げない。意味のあるアウトプットの後で一時停止する。
- 実行中に計画の `ゴール`、`アプローチ`、`成功基準` を変更しない。それらが変わる必要がある場合は、止まって計画し直す。
