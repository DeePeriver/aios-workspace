# Brainstorm

オーナーが質問を考え抜くのを助ける。選択肢を提示し、まだ1つに決めない。

## 手順

### 1. トピックを特定
質問を自分の言葉で言い換え、進む前にユーザーに確認する。これにより誤解を早期にキャッチする。

### 2. 関連するコンテキストを読み込む
当てはまるものだけ読む：

| トピック | 読むもの |
|---|---|
| 価格設定 | `customers.md`、`competitors.md`、`financials.md` |
| マーケティング / メッセージング | `customers.md`、`competitors.md`、`strategy.md` |
| 採用 / チーム | `personal-profile.md`、`financials.md`、`current-priorities.md` |
| プロダクト方針 | `customers.md`、`strategy.md` |
| 時間 / エネルギー / フォーカス | `personal-profile.md`、`current-priorities.md` |
| 戦略の転換 | `strategy.md`、`financials.md`、`current-priorities.md` |
| ベンダー / ツール | `financials.md`、`current-priorities.md` |

7つすべてを読まない。トピックがきれいに当てはまらない場合、どの次元が関連するかをユーザーに尋ねる。

### 3. 3〜5個の異なる角度を生成
各角度は：
- **本質的に異なる** — 同じアイデアの3つのフレーバーではない
- **読み込んだコンテキストに根ざしている** — コンテキストファイルの具体的事実を参照する
- **主要なトレードオフとセット** — このオプションは何を犠牲にする / リスクとするか？

### 4. 前提を問う
前提を問うか盲点を浮かび上がらせる質問を1〜2個提示する。例：
- 「Xが本当にここでの制約なのか考えたことはありますか？」
- 「オプションBが明らかな選択になるためには、何が真でなければなりませんか？」
- 「これがすでに試されていない理由はありますか？」

### 5. まだ推奨しない
ブレインストーミングは選択肢を提示する。ユーザーが推奨を求めた場合、`/aios-workspace:plan` で方向性にコミットしたいか尋ねる。

## 形式

会話的に保つ。ユーザーが書面のサマリーを求めない限り、見出しや箇条書きで構造化しすぎない。ゴールはレポートではなく考えるパートナー。最後に書面の成果物が欲しいとユーザーが言った場合、`outputs/brainstorms/YYYY-MM-DD-<slug>.md` に保存する。

## やってはいけないこと

- `context/` ディレクトリ全体を読み込まない。選択的に。
- `strategy.md` で明示的なもの（例：「選ばないこと」項目）に違反する選択肢を生成しない。生成しなかった場合はそれを言及する。
- 推奨に押し込まない。決定はユーザーが所有する。
